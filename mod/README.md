# Omega Client (companion mod - Fabric + Forge)

*The last client you will ever need.*

A small client mod that adds an in-game toggle menu for visual/QoL PvP settings, built for **both Fabric and Forge**. It's designed to be installed like any other mod through the Omega Client launcher's **Import your mods** button, and its config file (`config/omega-client.json`) is a plain JSON file the launcher's **Configure** button can already edit.

Open the menu in-game with **Right Shift** (rebindable in vanilla's Controls screen, under "Omega Client") - or click the small **Ω** button in the top-right corner of the vanilla pause menu (Esc), for anyone who hasn't found or rebound the key yet. Both open the exact same menu.

## Project layout

```
mod/
  common/    # Everything with no mapping-divergent type in its public API - see below for the full list
  fabric/    # Fabric build - mod id "omega-client"
  forge/     # Forge build - mod id "omega_client_forge" (Forge disallows hyphens in mod ids)
```

Both modules build through **Architectury Loom** rather than plain Fabric Loom / ForgeGradle (see "Architectury toolchain" below), which is what makes sharing `common/` source between a Yarn-mapped module and an officially-mapped one safe. Most feature logic that doesn't touch a mapping-divergent type directly (`ModConfig`, `SessionInfoLoader`, `SchematicStorage`, `FullbrightFeature`, `FovZoomFeature`, `ToggleSprintFeature`, `InfoHudFeature`, `ParticleFilter`, `CpsTracker`, `ParticleCategory`, `SchematicData`/`SchematicBlockEntry`, `SessionInfo`, `OmegaPresence`, plus the extracted `HighlightColorCache`/`BlockColorizer`/`LitematicaBitPacking` helpers - see below) now lives in `common/` and is compiled once. What's left duplicated in `fabric/`/`forge/` is duplicated **by necessity**, not by omission: Screens (extend Minecraft's own `Screen`), all Mixins (bytecode injection targets are inherently loader/mapping-specific), `PresenceNetworking` (Fabric's networking API vs. Forge's `SimpleChannel` are different systems), the `OmegaClient`/`OmegaClientForge` entrypoints (loader-specific lifecycle registration), and the handful of classes whose core algorithm operates directly on a mapping-divergent type - `BlockStateCodec` (`BlockState`), `SchematicSelection`/`SchematicCaptureFeature` (`BlockPos`/`World`/`Level`), `WireBoxRenderer` and the render-context parts of `BlockHighlightFeature`/`SchematicRenderFeature` (`VertexConsumer`/`MatrixStack`/`RenderLayer`/etc.), and the NBT-reading parts of `LitematicaImporter` (`NbtCompound`/`CompoundTag`). Each of those was checked for a self-contained pure-logic sliver first (see `HighlightColorCache`, `BlockColorizer`, `LitematicaBitPacking` below) before being left loader-specific.

### Architectury toolchain

Both `fabric/` and `forge/` build through `dev.architectury.loom` (+ the `architectury-plugin` for platform declarations) instead of plain `fabric-loom` / ForgeGradle. For Fabric this is close to a no-op - Architectury Loom is a drop-in superset of Fabric Loom, same `loom {}` extension and task outputs. For Forge it's a real toolchain swap: ForgeGradle and the separate `org.spongepowered.mixin` plugin are gone, replaced by Loom's own Mixin annotation processing (which auto-generates the refmap from the `.mixins.json` under `src/main/resources`, driven by the same `[[mixins]]` entry in `mods.toml` that already declared it at runtime) and the `forge(...)` dependency configuration in place of ForgeGradle's `minecraft(...)`.

Getting the Forge module's toolchain switch to actually compile took **seven** CI rounds, not one - each error was real and specific, but the fix wasn't obvious from any single error message alone. In order: missing `minecraft`/`mappings` dependencies; `forge(...)` still not found even with those added; `minecraft(...)` can't take a Forge coordinate directly (confirmed by Loom trying to look up a vanilla release literally named `1.20.1-47.2.0`); restructuring how `architectury-plugin`/`dev.architectury.loom` get applied (root `apply false` + subproject `apply plugin:`, matching a real reference mod); bumping Loom 1.6→1.7 (changed nothing, ruled out the version itself); adding an explicit `loom { forge { mixinConfig(...) } }` block (this one finally changed the *error*, not just failed the same way); and finally, per Loom's own diagnostic text, setting `loom.platform = forge` in a new `mod/forge/gradle.properties` - the actual root cause the whole time. `architectury { forge() }` declares platform intent to architectury-plugin's own model; it never flips Loom's internal Fabric/Forge switch, which is what actually controls whether `forge(...)` exists as a dependency method at all.

`common/` is now a real Architectury common module (not just a raw shared source directory): it compiles once against official mappings, and `fabric/`/`forge/` each pull its output via Loom's auto-generated `transformProductionFabric`/`transformProductionForge` configurations, merged into their final jars by the `shadow` plugin's `shadowJar`/`remapJar` wiring. This part came together in one CI-verified pass, modeled directly on the official Architectury multiloader template (`architectury/architectury-templates`) plus the same working reference mod (RoughlyEnoughItems) used to debug the Forge saga above.

Feature code has since moved into `common/` wherever it safely could: `ModConfig`, `SessionInfoLoader`, `SchematicStorage`, `FullbrightFeature`, `FovZoomFeature`, `ToggleSprintFeature`, `InfoHudFeature`, `ParticleFilter`, plus small pure-logic helpers extracted out of otherwise-loader-specific classes (`HighlightColorCache`, `BlockColorizer`, `LitematicaBitPacking`). The load-bearing rule discovered while doing this: **a method in `common/` can take or return primitives, records-of-primitives, or types already living in `common/` - never a raw Minecraft-mapped type (`GuiGraphics`, `Minecraft`, `KeyMapping`, `BlockPos`, `Identifier`/`ResourceLocation`, etc.) in its public signature.** `fabric/`'s compile-time view of `common/`'s classes is the *unremapped* official-mapped signature (the `common` dependency configuration, `namedElements`, is common's own native compiled output - only the final shadowJar-merged runtime artifact goes through `transformProductionFabric`'s remapping), so a parameter typed `GuiGraphics` there is a different, unrelated type from Fabric's own `DrawContext`-typed callback parameters, even though they're "the same" class once fully remapped at the bytecode level. `InfoHudFeature` hit this directly: its `tick()` and `render()` methods originally took `Minecraft`/`GuiGraphics` parameters and failed Fabric's compile with "cannot access GuiGraphics ... class file ... not found". The fix was to compute and cache only plain data in `common/` (calling `Minecraft.getInstance()` *internally*, which is fine - it's only the public signature that can't carry the type across) and keep the handful of lines that actually call `GuiGraphics`/`DrawContext` drawing methods as small per-loader glue in `OmegaClient.java`/`OmegaClientForge.java`, reading the cached state through plain getters. `ParticleFilter` hit the same wall from a different angle - `Identifier` and `ResourceLocation` differ in package *and* class name, so its public API takes extracted `String namespace, String path` instead, with the always-loader-specific Mixin caller doing the extraction. What's left in `fabric/`/`forge/` (Screens, Mixins, `PresenceNetworking`, the entrypoints, and the classes whose core algorithm operates directly on `BlockState`/`BlockPos`/`World`/`Level`/`VertexConsumer`/NBT types) stays duplicated because there's no more pure logic left to pull out without also pulling out a mapping-divergent type - see "Project layout" above for the full list.

This mod is otherwise **zero-Mixin by design** - every other feature reads/writes plain client-option fields or hooks a documented Fabric API / Forge event, deliberately avoiding the extra fragility of bytecode-injection tooling in a project with no way to compile-verify it. Particle control (see below) is the one place that turned out to be genuinely impossible without it, and is called out everywhere it matters.

Both modules now **compile clean against real Minecraft classes in CI** (see "Building" below) - the Fabric module on its first attempt, the Forge module after two rounds of corrections that all landed on pre-flagged spots. Compiling isn't the same as runtime-tested, but the "wild guess" phase of the Forge port is over.

## What it does - and deliberately doesn't

Every feature here is a visual or convenience setting, nothing that reads information the player couldn't already get, and nothing that automates input:

| Feature | What it actually does |
|---|---|
| **Fullbright** | Raises the gamma option past vanilla's UI-enforced 0-1 slider range. Purely a brightness override, same technique used by countless QoL mods. |
| **Block Highlight** | Draws a wireframe outline around configured block types (defaults: obsidian, respawn anchor, crying obsidian) within 20 blocks. Rendered with the same depth-tested "lines" layer vanilla's own F3+B hitbox debug view uses - **terrain still occludes it**. This is "combat clarity" (see anchors/crystals you're already looking near more easily), not X-ray: it will never show a block through a solid wall. The target block list and outline color are both editable in-game too, via **Visual Settings...**. |
| **Custom FOV / Zoom** | Same no-mixin technique as fullbright, applied to the FOV option. Zoom is hold-to-zoom via a keybind (default **C**). The on/off switch lives on the main menu; the actual FOV and Zoom FOV numbers are set in-game via **Visual Settings...**. |
| **Toggle Sprint** | Keeps you sprinting while holding forward, so you don't need double-tap or a sprint-lock keybind from another mod. Doesn't change movement speed or add anything beyond vanilla sprinting. |
| **Info HUD** | Coordinates, FPS, ping, facing direction, a CPS (clicks-per-second) counter, and a WASD+space+click keystroke display - each line individually toggleable in-game via **HUD...** (the master on/off switch lives on the main menu). All of this is already visible via vanilla's F3 debug screen or tab list - this is just an always-on, friendlier subset of it. The CPS counter *reads* your click rate (sampled per-frame, since fast clicks are shorter than a game tick); it never generates clicks. |
| **No Hurt Camera** | Cancels the camera tilt/shake when you take damage. Purely visual comfort - damage, knockback and the red flash are untouched. |
| **No Fog** | Pushes the fog planes out to infinity after vanilla sets them up each frame - terrain, water, lava and nether fog alike. Doesn't extend render distance or reveal anything vanilla wouldn't render anyway. |
| **Clear Weather** | Client-visual only: reports a zero rain/thunder gradient on the client world, so you never see or hear rain. Guarded to never touch the integrated server's real weather - crops, mob spawning, tridents and lightning all behave exactly as the server decides. |
| **Weather & Time changer** | A "Weather & Time..." menu with buttons for Day/Noon/Night/Midnight and Clear/Rain/Thunder. Unlike Clear Weather above, these send the *real* vanilla `/time set` and `/weather` commands - the same as typing them in chat - so they only do anything where you already have permission to run them (singleplayer, or op'd on a server). No permission is granted that the player didn't already have; a server that rejects `/weather clear` from chat rejects it from this button too. |
| **Schematics** | A WorldEdit-style two-point selection, save-to-file, and ghost-preview tool, plus a best-effort Litematica `.litematic` file importer - see below. |
| **Omega presence badge** | An Ω (brand red) rendered left of the nametag of players known to be on Omega Client. Backend-less design ceiling, stated plainly: a client cannot know what another client runs on a plain vanilla server - presence is announced over a custom `omega-client:presence` payload channel, which is silently dormant unless the server/proxy relays it; a future hosted presence API can feed the same set (`common/presence/OmegaPresence`). Paid cosmetics ride the same channel: a badge kind recolors the Ω; gear kinds (hat/cape/wings) render one of two ways, per cosmetic. PROCEDURAL (most of them) is pixel art (`common/presence/CosmeticPixelArt`) meshed onto the wearer's model by `common/presence/CosmeticGeometry` - flat art extruded like a Minecraft item texture (each opaque pixel a colored cell, transparency cutting the silhouette), or, for hats (e.g. `azure_charm_hat`), true 3D voxel art: layered top-down slices face-culled into a real dome/brim volume, the same multi-box construction Lunar/Feather/Essential-style hats use. TEXTURED (capes only so far, e.g. `starlit_cape`) wraps a real PNG onto 8 cloth-like horizontal strips (`common/presence/CosmeticTexturedMesh`) via an actual Minecraft textured render type - the same technique vanilla uses for its own player cape - letting a cosmetic carry real gradients/glow a coarse pixel grid can't represent. Both are drawn via Fabric API's feature-renderer callback / Forge's `AddLayers` event, no extra Mixins (`fabric/render/CosmeticFeatureRenderer`, `forge/render/CosmeticRenderLayer`). Cape and wing geometry sways/flaps every frame - a stylized procedural rotation about each cosmetic's shoulder/collar pivot, scaled by distance from that pivot and by the wearer's vanilla limb-swing-amount (`common/presence/CosmeticAnimation`), not a physics or cloth simulation; hats stay rigid by design; a TEXTURED cape's 8 strips each rotate individually so the texture bends smoothly rather than swinging as one rigid slab. Capes/wings can also opt into a sparse colored particle trail from their tip (`common/presence/CosmeticTrail` for the pure world-placement math, `DustParticleEffect`/`DustParticleOptions` in the two renderers for the actual spawn) - an authoring choice per cosmetic, not automatic, and works the same for either rendering style. Toggleable (default on, one switch covers badge + gear). |
| **Cosmetics visibility** | A "Cosmetics..." menu with a master on/off switch, a "My cosmetics" / "Other players' cosmetics" split, and a per-kind toggle (Hats/Capes/Wings) - all default on. Purely controls whether `CosmeticFeatureRenderer`/`CosmeticRenderLayer` draw a wearer's cosmetic at all; separate from the Ω badge toggle above and from which cosmetic is equipped (a single `ownedCosmeticId`, set by the launcher's Cosmetics page redeem flow). |
| **Particle control** | Per-category on/off switches (block, ambient block, totem, crit, explosion, portal), a free-form blacklist for anything else, and a density slider to thin out whatever's left. Purely visual/performance tuning, same spirit as vanilla's own Particles option, just more granular - see below. |

**Intentionally not included**, because they cross from "visual setting" into "unfair advantage against other players" and most servers treat them as cheating: reach/hitbox expansion, aimbot/kill-aura, auto-clicking, velocity or knockback modification, X-ray (seeing blocks *through* terrain), and anything that reads server-side information the client wouldn't normally have.

Using visual settings like fullbright on a server that specifically disallows them can still get you moderated - check the rules of wherever you're playing.

## Schematics

A small WorldEdit/Litematica-style building tool: select a region, save it, then load it back later as a ghost-preview overlay to build against. Works identically on both loaders, and `.omschem.json` files are interchangeable between the two builds (same format, same folder location).

1. Look at one corner block and press **O** (Position 1), then the opposite corner and press **P** (Position 2). Both get an action-bar confirmation.
2. Open the menu (**Right Shift**) → **Schematics...**, type a name, and hit **Save Selection**.
3. To build from it later: open **Schematics...**, click the saved name to load and start previewing it, stand where you want its corner to land, and click **Re-anchor to Me**.

**Saving does not write the real Litematica `.litematic` file format.** That format is proprietary, undocumented binary NBT (bit-packed per-block-state arrays, sub-region support, etc.), and this project has no way to verify byte-for-byte *write* compatibility with it without a real game session to test against - getting it subtly wrong would produce files that silently don't work in actual Litematica. Instead, saving/previewing uses Omega Client's own format ("Omega Schematic", `.omschem.json` - plain, human-readable JSON via Gson) that does the same *job* without claiming write-compatibility it can't back up. Files live in `<config>/omega-client/schematics/`. Full block state (orientation, waterlogged, stair shape, etc.) is captured, not just the block type - see `BlockStateCodec.java` in each loader module.

**Importing real `.litematic` files is supported, best-effort.** Drop `.litematic` files into `<config>/omega-client/schematics/import/` and they'll show up as "Import: filename.litematic" buttons in the Schematics screen; clicking one converts and saves it as a normal Omega Schematic. Unlike everything else in this mod, `LitematicaImporter.java` isn't built on documented/stable Minecraft APIs - it's reconstructed from memory of how the Litematica format is commonly described (community write-ups, other third-party readers), since the format itself is undocumented and this project has no real `.litematic` file to test against. The bit-unpacking *arithmetic* is verified correct via an isolated round-trip simulation (shift/mask math has no bugs), but whether it matches Litematica's *actual* on-disk layout is unverified. Concretely:
- Only the first region in a file is imported (Litematica's multi-region support is out of scope here).
- A failed/wrong-looking import shows an error message rather than crashing your game (defensive by design, given the uncertainty above) - always eyeball an imported schematic's preview before building from it.
- If it doesn't work for your file, the most likely culprits are the exact `BlockStatePalette`/`BlockStates` NBT tag names, or Litematica having changed its bit-packing scheme between versions.

**The ghost preview is color-coded wireframe outlines, not real block textures.** Textured block rendering exists in the Minecraft client but calling it correctly is another spot this project can't compile-verify, so the preview reuses the same depth-tested wireframe technique as Block Highlight instead (each exact block state gets a consistent deterministic color, so different materials/orientations are still distinguishable) - see `SchematicRenderFeature.java` in each loader module.

## Account switching

The menu's header shows "Playing as: {username} ({microsoft|offline})", read from a small JSON session file (`omega-client-session.json`) the Omega Client launcher writes into the game directory right before spawning the game - see `SessionInfo.java` in `common/` and each loader's `SessionInfoLoader.java`.

Minecraft has no API for hot-swapping a live account mid-session, so **"Switch Account" doesn't actually switch anything in-game** - clicking it (twice, to confirm) writes a marker file (`omega-client-switch-account.request`) into the game directory and then quits the client (`MinecraftClient.scheduleStop()` on Fabric, `Minecraft.stop()` on Forge - a clean, same-tick shutdown request, not `System.exit()`). The launcher watches for that marker file after its spawned game process exits and, if present, brings itself to the foreground and reopens its own account switcher automatically. Both sides only ever communicate through these two files - there's no socket or IPC channel between the launcher and a running game.

## Particle control

Open the menu (**Right Shift**) → **Particles...** for:
- **A master switch** - turns every particle off, full stop. Two Mixin injections back this, not one: the normal type-classified spawn path (`addParticle(ParticleEffect, ...)` / `createParticle(ParticleOptions, ...)`), plus a second, category-blind injection on the lower-level `addParticle(Particle)` / `add(Particle)` overload that a handful of vanilla effects use to spawn already-constructed child particles directly (bypassing type lookup entirely, e.g. some particles spawning their own children from `tick()`). Only the master switch can act on that second path - a raw `Particle` object doesn't carry the type info a category check needs - but it's covered, so "All particles: OFF" actually means all of them, not "most, from the normal spawn path." Turning it off stops new particles from appearing immediately; particles already alive at that moment still finish their remaining lifetime rather than vanishing instantly (typically under a second, up to a few seconds for slow ones like smoke).
- **Six category switches** - Block (the generic break/step/land particle - "particles for every block"), Ambient block (torch smoke, drips, spores, and similar decorations), Totem, Crit (crit + enchanted-hit), Explosion, and Portal.
- **A custom blacklist** - type any particle id (e.g. `minecraft:soul`, or just `soul` and it'll assume `minecraft:`) to block something the categories above don't cover - the "and more" from the original request.
- **A density slider** (cycles 100% → 75% → 50% → 25% → 10%) - probabilistically thins whatever's still allowed through, instead of an all-or-nothing cut.

Settings are per-category, not per-block - "turn off particles for every block" means the Block/Ambient switches, not a thousand individual block toggles. Use the blacklist for anything more specific.

**This is the mod's one deliberate exception to being otherwise zero-Mixin.** There's no documented Fabric API event or Forge event for cancelling an individual particle spawn by type, and every non-Mixin alternative considered (replacing the client's particle-manager instance, or reflectively wrapping its registered particle factories) is strictly more fragile - both depend on reproducing or racing against internal state that gets rebuilt on every resource pack reload. A single injection at the front of the one real choke point - every particle spawn in the game funnels through `ParticleManager#addParticle` (Fabric) / `ParticleEngine#createParticle` (Forge) - is the smaller, more standard risk; it's a well-established target for real particle-control mods, not a novel hack. See `ParticleManagerMixin.java` (Fabric) and `ParticleEngineMixin.java` (Forge).

The classification itself (which category a given particle id falls into) is pure string matching in `common/ParticleCategory.java` and is deliberately conservative - unrecognized ids (including anything from a modded namespace) fall through as uncategorized and are only affected by the master switch, blacklist, and density slider, never silently lumped into the wrong category. The per-spawn check is written to allocate nothing in the common case (master on, no blacklist, density 100%) since it runs on every particle, every frame - see the hot-path note in `ParticleFilter.java`.

On the Fabric side, adding this Mixin needed no extra Gradle wiring - Fabric Loom bundles Mixin annotation processing (and refmap generation) already, the mod just needed a `.mixins.json` and a `"mixins"` entry in `fabric.mod.json`. Forge originally needed real Gradle changes for the same thing - an `annotationProcessor` dependency on SpongePowered's Mixin plus the separate MixinGradle plugin to generate the SRG refmap, since Forge 1.20.1 runs SRG names in production. An earlier version of that setup tried to skip refmap generation with a compiler flag; the first real CI compile rejected that immediately, which is exactly the kind of correction the risk-flagging in this README exists to invite. Since the Architectury Loom migration, Forge gets the same free Mixin AP + refmap generation Fabric always had - the SpongePowered plugin and its explicit refmap wiring are gone, and the `[[mixins]]` entry in `mods.toml` is now the only thing declaring this Mixin, same as before.

## Building

**Both loader modules now compile clean against real Minecraft/Fabric/Forge classes** - verified by the repo's GitHub Actions workflow (`.github/workflows/build.yml`), which produces both jars as downloadable artifacts on every push. The Fabric module compiled on its very first real-classpath attempt; Forge needed two rounds of fixes, every one landing on a spot the lists below had pre-flagged (the particle Mixin's method signature/refmap wiring, `Options.gamma`/`fov` accessors, `Property.getName`/`getValue`).

What remains genuinely unverified is **runtime behavior**: compiling proves the API names and signatures are right, not that a feature behaves correctly in a live game session (rendering looks right, keybinds feel right, the Litematica importer parses real files). The "spots to check" lists below are kept for that reason - they're now ranked evidence of what was risky, and the remaining runtime-only risks (especially `LitematicaImporter.java`) still stand.

Build one or both:

```bash
cd mod
gradle :fabric:build     # -> fabric/build/libs/omega-client-fabric-0.1.0.jar
gradle :forge:build      # -> forge/build/libs/omega-client-forge-0.1.0.jar
```

No Gradle wrapper is checked in (the sandbox that generated this project couldn't validate the wrapper's distribution URL through its proxy) - use a local Gradle install, or open the project in IntelliJ IDEA with the Minecraft Development plugin, which sets one up automatically.

Requirements:
- JDK 17+
- Network access to `maven.fabricmc.net` + Mojang's piston-meta (Fabric), or `maven.minecraftforge.net` + Mojang's piston-meta (Forge)

### Fabric: spots to check first if it doesn't compile

Roughly in order of how likely they are to have drifted:
- `LitematicaImporter.java` - the whole file, undocumented third-party format, see "Schematics" above.
- `Property.name(value)` / `Property.parse(text)` and `StateManager.getProperty(name)` in `BlockStateCodec.java`.
- `NbtIo.readCompressed(InputStream)` in `LitematicaImporter.java` - some MC versions require an extra `NbtSizeTracker` argument.
- `GameOptions.getGamma()` / `getFov()` in `FullbrightFeature.java` / `FovZoomFeature.java`.
- `ClientPlayerEntity.sendMessage(Text, boolean)` in `OmegaClient.java`.
- `MinecraftClient.scheduleStop()` in `ClickGuiScreen.java` (account switching) - this one's lower-risk than most of this list, it's a long-standing Yarn name with no known history of changing.
- `EntityRenderer#renderLabelIfPresent`'s @ModifyVariable target in `EntityRendererMixin.java` (presence badge) - and the Fabric networking pair (`ClientPlayNetworking` receiver + `ClientPlayConnectionEvents.JOIN` sender) in `network/PresenceNetworking.java`.
- `ParticleManager#addParticle(ParticleEffect, double,double,double,double,double,double)` and `ParticleManager#addParticle(Particle)`'s exact method descriptors in `ParticleManagerMixin.java` (particle control) - if either signature has drifted, Mixin fails loudly at startup (the config is `"required": true`) rather than silently doing nothing, so a broken match is easy to notice.
- `GameRenderer#tiltViewWhenHurt` in `GameRendererMixin.java` (No Hurt Camera) and `BackgroundRenderer#applyFog` in `BackgroundRendererMixin.java` (No Fog) - both handlers deliberately capture no arguments, so only the method *names* have to be right, not the parameter lists. `World#getRainGradient`/`getThunderGradient` + the public `isClient` field in `WorldMixin.java` (Clear Weather) are the same class of guess.
- `PlayerListEntry.getLatency()` via `getNetworkHandler().getPlayerListEntry(uuid)`, `Direction.getName()`, and `Window.getHandle()` in `InfoHudFeature.java` (ping / facing / CPS lines).
- `ClientPlayNetworkHandler.sendChatCommand(String)` in `WeatherTimeScreen.java` (weather/time changer) - a fresh guess, though a low-risk one: it takes a single `String` argument and captures nothing else, so there's no parameter-shape ambiguity, just the method name itself.

Your IDE's autocomplete on `client.options.`, `player.sendMessage(`, `block.getStateManager().`, or `NbtIo.` will show the real signatures for your exact `yarn_mappings` build if any of these have drifted.

### Forge: spots to check first if it doesn't compile

This list is longer than Fabric's because every file in `forge/` is a fresh, never-checked translation from Yarn names to Mojang's official mappings - not a difference in the underlying logic, just more surface area that's never touched a real compiler:

- **`OmegaClientForge.java`'s event wiring** - the highest-risk spot in the whole module. Forge's client rendering and HUD-overlay APIs changed shape more than once across the 1.20.x line. `RegisterGuiOverlaysEvent`/`event.registerAboveAll(...)`, `RenderLevelStageEvent`/`Stage.AFTER_TRANSLUCENT_BLOCKS`, and `TickEvent.ClientTickEvent` are all reasonable guesses, not verified ones.
- **`build.gradle`'s ForgeGradle DSL** (`minecraft { mappings channel: 'official', ... }`, the `runs { client { ... } }` block, the `[6.0,6.2)` plugin version range) - the standard MDK-template shape, but unverified against the actual plugin.
- **Field-vs-getter guesses everywhere a Fabric file used a Yarn getter** - e.g. `options.gamma`/`options.fov` as direct fields (`FullbrightFeature.java`, `FovZoomFeature.java`) instead of Yarn's `getGamma()`/`getFov()`. Official mappings generally expose these as plain fields, but the exact field names are a guess.
- **Movement/action keybinding field names** in `ToggleSprintFeature.java` and `InfoHudFeature.java` (`options.keyUp`, `keyLeft`, `keyDown`, `keyRight`, `keyJump`, `keyAttack`) and the sneak/hunger renames (`isShiftKeyDown()`, `getFoodData()`).
- **`BlockStateCodec.java`** - the same `Property.name()`/`parse()` moderate-confidence spot as Fabric's, plus fresh guesses at `BuiltInRegistries`, `StateDefinition`, `defaultBlockState()`, `setValue()`.
- **`player.displayClientMessage(Component, boolean)`** in `OmegaClientForge.java` - the official-mappings equivalent of Fabric's `sendMessage(Text, boolean)`.
- **`.bounds(x, y, w, h)`** on `Button.Builder` in `ClickGuiScreen.java`/`SchematicScreen.java` - a guess at the official-mappings equivalent of Yarn's convenience `.dimensions(...)` call.

## Multi-version architecture

Long-term goal: support many Minecraft versions at once, the way Lunar/Essential-style clients do - the launcher downloads the right pre-built jar per instance automatically (`bundledMods.ts`'s `OMEGA_MOD_MINECRAFT_VERSION` gate is the launcher-side half of this; it currently skips any instance that isn't the one built version rather than shipping a mismatched jar that would abort the launch). Reaching that means the mod itself has to be portable across versions without a rewrite each time. The approach is the standard "core vs. injection points" split, layered on top of the `common/` vs. `fabric/`/`forge/` split that already exists for the *loader* axis:

- **Core** - `common/`. Feature logic that never names a version-volatile Minecraft symbol. Already the bulk of the mod (`ModConfig`, `FullbrightFeature`, `InfoHudFeature`, `ParticleFilter`, `CosmeticCatalog`, `OmegaPresence`, schematic data, ...). Compiled once, remapped per loader.
- **Injection points** - the Mixins in `fabric/*/mixin` and `forge/*/mixin`. These are the *only* files that inherently name obfuscated methods/classes the mappings rename between versions (e.g. No-Hurt-Cam targets Yarn `tiltViewWhenHurt` / Mojmap `bobHurt`), so they're what a new Minecraft version actually forces you to re-author. They now hold **no decision logic of their own** - each one extracts the plain values it needs from the Minecraft objects it was handed, calls into `common/platform/OmegaHooks` for the decision, and applies the result back. That keeps every real decision in one deduplicated place and shrinks each injection point to just its version-specific coordinate.

`OmegaHooks` is the seam between the two. Its one hard rule - **methods take and return only cross-mapping-safe types (JDK types + mod-owned types), never a Minecraft class** - is what lets a Yarn-mapped Fabric mixin and a Mojmap-mapped Forge mixin call the exact same method (this is the same `common/`-public-API constraint documented under "Architectury toolchain" above, applied to the hook direction). The nametag badge is the clearest payoff: the "is this a known Omega user, what color" decision was byte-for-byte duplicated across the two `EntityRendererMixin`s and is now `OmegaHooks.nametagBadgeColor(UUID)` called from both, leaving each mixin with only its loader's `Text`/`Component` wrapping.

**Not yet done** (this is the foundation, not the finished system): a per-version source axis (e.g. `fabric-1.20.1/` / `fabric-1.21.1/` overlays holding just that version's mixins, everything else shared), a CI build matrix producing version-tagged jars, and the launcher-side change to `omegaJarFor()` picking the jar matching an instance's resolved Minecraft version instead of one fixed build. Those are additive on top of this seam - adding a version becomes "write a new small mixin set implementing the known hook coordinates," not touching core, screens, config, networking, or cosmetics.
